# Design Spec: Canary Release Workflow

**Status:** Draft
**Date:** 2026-03-13
**Related ADR:** `adr_release_install_strategy.md` (Section 4: Pre-Release / Canary Workflow)

## Problem Statement

The current `deploy-canary.yml` conflates two concerns: deep verification (cross-platform builds + tests) and canary release publishing. The refactoring splits these into:

1. `verify-deep.yml` -- full test suite, no release artifacts (handled separately)
2. A new canary release workflow -- produces pre-release binaries using cargo-dist infrastructure

The canary release needs to build the same 8-target matrix as the real release, produce the same archive formats (`.tar.xz` for Unix, `.zip` for Windows), and publish them under a single overwritten `canary` pre-release tag. It must NOT duplicate the cargo-dist build matrix by hand.

## Analysis

### Option 1: `workflow_call` Reusable Workflow from `release.yml`

Extract the build jobs from `release.yml` into a reusable workflow, then call it from both `release.yml` and the canary workflow.

**Pros:**
- Single source of truth for the build matrix
- Changes to targets propagate automatically

**Cons:**
- `release.yml` is **auto-generated by cargo-dist** (`dist init` / `dist generate`). Any structural changes will be overwritten the next time cargo-dist regenerates the workflow. This is the fundamental blocker -- cargo-dist owns `release.yml` and expects to control its structure entirely.
- The `plan` job produces a dynamic matrix from `dist plan --output-format=json`. This matrix is tightly coupled to cargo-dist internals (runner selection, container images, dist-args, install-dist expressions). Extracting it into a reusable workflow would require either (a) running `dist plan` in the canary workflow too (which requires a tag), or (b) hardcoding the matrix (defeating the purpose).
- cargo-dist's `host --steps=create` creates the GitHub Release and expects a real semver tag. The canary workflow uses a non-semver `canary` tag.

**Verdict:** Not viable without forking cargo-dist's workflow generation.

### Option 2: Call `cargo-dist` CLI Directly in the Canary Workflow

Use the `dist build` CLI command inside a custom workflow with a manually maintained matrix that mirrors cargo-dist's targets. Let cargo-dist handle archive creation, checksums, and naming conventions, but manage the GitHub Release separately.

**Pros:**
- cargo-dist handles archive format, naming, checksums -- consistent with real releases
- No modifications to `release.yml` needed
- The canary workflow is fully independent and does not interfere with cargo-dist's generated workflow

**Cons:**
- The build matrix must be maintained in two places (`dist-workspace.toml` targets and the canary workflow matrix). However, targets change very rarely (last change was the initial setup), and the canary workflow can include a comment pointing to `dist-workspace.toml` as the source of truth.
- `dist build` without a tag requires `--tag` to be synthesized or the `--artifacts=local` flag to be used without tag context. Testing needed to confirm `dist build` works without a real git tag.

**Verdict:** This is the pragmatic choice, but `dist build` is designed around tags. It may not work cleanly without one.

### Option 3: Custom Build Matrix with cargo-dist Archive Tooling

Build binaries using the same `actions-rust-cross` and native cargo approaches as the current `deploy-canary.yml`, but use `dist` only for the archive/checksum step (or skip dist entirely and use `tar`/`zip` directly).

**Pros:**
- Full control over the workflow; no cargo-dist CLI dependency at build time
- Straightforward -- the current `deploy-canary.yml` already does this (minus archive formatting)
- No tag required

**Cons:**
- Archive naming and format must be maintained manually (though the convention is simple: `ocx-{target}.tar.xz` / `.zip`)
- No `dist-manifest.json` produced (not needed for canary)

**Verdict:** Simplest and most robust. The canary release does not need cargo-dist's manifest system or its complex plan/build/host pipeline.

### Option 4: Lightweight Tag + Full cargo-dist Pipeline

Create a temporary lightweight tag (e.g., `v0.0.0-canary.{run_id}`), run the full `release.yml` pipeline against it, then delete the tag and move the `canary` tag to point at the release.

**Pros:**
- True reuse of cargo-dist pipeline
- Identical build artifacts

**Cons:**
- Pollutes git tag namespace (even if cleaned up, race conditions exist)
- `release.yml` runs post-release jobs (version bump PR) that must be suppressed
- Fragile -- depends on timing of tag creation/deletion
- The `verify-version` plan job would fail because Cargo.toml version won't match the synthetic tag

**Verdict:** Too complex and fragile.

## Recommendation: Option 2 -- dist CLI with Canary Version (Implemented)

The canary workflow uses `dist plan` and `dist build` directly, with a computed canary prerelease version. The plan job computes a version like `0.6.0-canary.b4a845b` (bumped version + short SHA), sets it in Cargo.toml temporarily, and runs `dist plan --tag=v{version}` to produce the dynamic 8-entry build matrix. Build jobs mirror release.yml's `build-local-artifacts` and `build-global-artifacts` structure exactly, with a simple `sed` to set the canary version before building. A custom `release` job handles changelog generation and the `canary` pre-release tag.

**Rationale:**
- DRY: the build matrix, runners, cross-compilation strategy, archive creation, and checksums all come from dist via `dist-workspace.toml`. No manual matrix to maintain.
- The canary workflow does not modify `release.yml` -- it mirrors the structure independently.
- dist's prerelease detection (`-canary.` suffix) automatically marks `announcement_is_prerelease: true`.
- The embedded binary version (`ocx version`) shows the canary version, useful for debugging.

## Workflow Design

### Trigger

```yaml
on:
  push:
    branches: [main]
  workflow_dispatch:
```

The workflow runs on every push to `main` and can be triggered manually. No concurrency control is needed beyond GitHub's default (but a concurrency group should cancel in-progress runs when a new push arrives).

```yaml
concurrency:
  group: canary-release
  cancel-in-progress: true
```

### Job Structure

```
push to main
    |
    v
build (matrix: 8 targets)
    |
    v
release (fan-in: collect artifacts, create/update canary pre-release)
```

Two jobs total. No tests, no verification, no post-release steps.

### Build Job

A matrix of 8 targets, split into native and cross builds:

| Target | Runner | Method |
|--------|--------|--------|
| `x86_64-unknown-linux-gnu` | `ubuntu-latest` | Native (`cargo build`) |
| `aarch64-unknown-linux-gnu` | `ubuntu-latest` | Cross (`actions-rust-cross`) |
| `x86_64-unknown-linux-musl` | `ubuntu-latest` | Cross (`actions-rust-cross`) |
| `aarch64-unknown-linux-musl` | `ubuntu-latest` | Cross (`actions-rust-cross`) |
| `x86_64-apple-darwin` | `macos-latest` | Cross (`actions-rust-cross`) |
| `aarch64-apple-darwin` | `macos-latest` | Native (`cargo build`) |
| `x86_64-pc-windows-msvc` | `windows-latest` | Native (`cargo build`) |
| `aarch64-pc-windows-msvc` | `windows-latest` | Cross (`actions-rust-cross`) |

The existing `.github/actions/build-rust` action handles native builds. For cross builds, `houseabsolute/actions-rust-cross` is already used in the current canary workflow.

Each build job:
1. Checks out the repo (with submodules)
2. Builds `--release --locked` for the target
3. Creates an archive: `.tar.xz` for Unix targets, `.zip` for Windows targets
4. Generates a SHA-256 checksum for the archive
5. Uploads the archive and checksum as an artifact

**Archive naming convention** (matches cargo-dist output):
- `ocx-aarch64-apple-darwin.tar.xz`
- `ocx-x86_64-pc-windows-msvc.zip`
- etc.

**Archive contents:** The binary (`ocx` or `ocx.exe`) at the root of the archive. No nested directories -- this matches cargo-dist's default layout and is what the install scripts expect.

### Release Job

Depends on all build jobs completing. Steps:

1. Checkout (for `git-cliff`)
2. Run `git-cliff --unreleased --strip header` to generate the changelog since the last release tag
3. Download all build artifacts
4. Concatenate individual checksum files into a single `sha256sum.txt`
5. Delete the existing `canary` release (if it exists) using `gh release delete canary --yes`
6. Delete the existing `canary` tag (if it exists) using `gh tag delete canary` (remote) or `git push --delete origin canary`
7. Create a new pre-release with `softprops/action-gh-release`:

```yaml
- uses: softprops/action-gh-release@v2
  with:
    tag_name: canary
    name: "Canary (main@${{ github.sha }})"
    body: |
      **Canary build from `main`**

      | | |
      |---|---|
      | **Commit** | [`${{ github.sha }}`](${{ github.server_url }}/${{ github.repository }}/commit/${{ github.sha }}) |
      | **Date** | ${{ github.event.head_commit.timestamp }} |
      | **Triggered by** | @${{ github.actor }} |

      ---

      ## Changes since last release

      ${{ steps.cliff.outputs.content || 'No conventional commits since last release.' }}
    prerelease: true
    make_latest: false
    files: |
      artifacts/*.tar.xz
      artifacts/*.zip
      artifacts/sha256sum.txt
```

### Tag Management

The `canary` tag is a moving target -- it gets re-pointed to the latest `main` commit on each run. The `softprops/action-gh-release` action handles this: if the tag already exists, it updates the release and moves the tag. However, to avoid stale artifacts accumulating on the release, the workflow should delete the existing release first and recreate it (rather than updating in place).

The sequence is:
1. `gh release delete canary --yes --cleanup-tag` (deletes release AND tag)
2. `softprops/action-gh-release` creates fresh release + tag pointing at `${{ github.sha }}`

This avoids the problem where updating a release retains old artifacts from previous runs.

### Permissions

```yaml
permissions:
  contents: write
```

Required for creating releases and tags. No other permissions needed (no test result publishing, no PR creation).

## Changes Required to `release.yml`

**None.** The canary workflow is fully independent of the cargo-dist release workflow. This is a key design goal -- cargo-dist can regenerate `release.yml` at any time without affecting canary builds.

## Archive Creation

Each build job creates its own archive. The approach differs by platform:

**Unix targets (`.tar.xz`):**
```bash
mkdir -p staging
cp "target/${TARGET}/release/ocx" staging/
tar -cJf "ocx-${TARGET}.tar.xz" -C staging ocx
sha256sum "ocx-${TARGET}.tar.xz" > "ocx-${TARGET}.tar.xz.sha256"
```

**Windows targets (`.zip`):**
```powershell
New-Item -ItemType Directory -Path staging
Copy-Item "target\${TARGET}\release\ocx.exe" staging\
Compress-Archive -Path staging\ocx.exe -DestinationPath "ocx-${TARGET}.zip"
(Get-FileHash "ocx-${TARGET}.zip" -Algorithm SHA256).Hash.ToLower() + "  ocx-${TARGET}.zip" | Out-File "ocx-${TARGET}.zip.sha256" -Encoding ascii
```

The individual `.sha256` files are merged in the release job:
```bash
cat artifacts/*.sha256 > artifacts/sha256sum.txt
rm artifacts/*.sha256
```

## Trade-offs

| Decision | Trade-off |
|----------|-----------|
| Custom matrix (not reusing cargo-dist) | Must manually sync if targets change. Targets have changed once in project history. Low risk. |
| No tests in canary | Canary may publish broken binaries. Mitigated by `verify-deep.yml` being run before merging to main, and by the `canary` tag being clearly marked as pre-release. |
| Delete + recreate release (not update) | Brief window where `canary` release does not exist. Acceptable for a pre-release. Prevents artifact accumulation. |
| `push to main` trigger (not just `workflow_dispatch`) | More CI minutes consumed. Mitigated by `cancel-in-progress` which aborts stale runs. |
| Archive format matches cargo-dist | Binary at archive root (no directory nesting). Keeps install scripts working with both canary and release archives. |

## Risks

1. **Archive format drift.** If cargo-dist changes its archive layout in a future version, canary and release archives may diverge. Mitigation: the archive format is simple (single binary at root) and unlikely to change. Add a comment in the workflow pointing to `dist-workspace.toml` as the reference.

2. **Concurrent runs on rapid pushes.** The `cancel-in-progress` concurrency group handles this, but if a release job is mid-upload when cancelled, the `canary` release could be in a deleted state. Mitigation: `cancel-in-progress` only cancels the older run, and the newer run will recreate the release.

3. **`gh release delete` failure on first run.** If no `canary` release exists yet, the delete command will fail. Mitigation: use `|| true` to ignore the error.

## Implementation Steps

1. Create `.github/workflows/canary-release.yml` with the workflow defined above
2. Remove the `snapshot-release` and `build-cross` jobs from `deploy-canary.yml` (those concerns move to `canary-release.yml` and `verify-deep.yml` respectively)
3. Rename or delete `deploy-canary.yml` once `verify-deep.yml` and `canary-release.yml` are both in place
4. Test by pushing to `main` and verifying:
   - All 8 targets build successfully
   - Archives contain the binary at the root
   - `sha256sum.txt` is present and correct
   - Release body contains commit SHA, date, actor, and changelog
   - Release is marked as pre-release, not latest
5. Verify that `release.yml` continues to work unchanged for tagged releases
