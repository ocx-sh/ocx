## Objective

Add `ocx shell profile` subcommands that persist installed tools into a `$OCX_HOME/profile.json` manifest. The env file delegates to `ocx shell profile load` which reads the manifest and outputs shell-specific exports at shell startup, making profiled tools available in every new shell session.

## Scope

### In Scope

- `$OCX_HOME/profile.json` manifest with `version: 1` field and JSON schema generation
- `ocx shell profile add/remove/list/load` subcommands
- Support for `--candidate` and `--current` resolution modes per entry
- Updated `install.sh`, Fish config, and `install.ps1` env file templates with `shell profile load` eval line
- Unit tests and acceptance tests

### Out of Scope

- Multiple named profiles (future — data model supports it)
- `--profile` flag on `install`/`uninstall` commands
- Automatic profile update on `ocx select`/`ocx uninstall`

## Design Validation

Industry research confirms the dynamic eval approach is sound:

- **Comparable tools** (fnm, direnv) use the same pattern with Rust/Go binaries and report 5-15ms startup — well under the 50ms "unnoticeable" threshold.
- **`Context::try_init` with `--offline`** skips OCI client creation — just struct assembly and path computation. The `shell profile load` command then reads `profile.json` (1 read) + `metadata.json` per package (N reads) + string formatting. All in-process, no additional subprocess spawns per package.
- **No per-prompt hook** (unlike mise/direnv) — cost is one-time per shell session, not per keystroke.
- **Tools that get complaints** are slow interpreters (nvm/bash at 300-500ms, Homebrew/Ruby at 20ms) or per-prompt hooks. OCX does neither.

## Technical Approach

### Architecture Changes

```
~/.ocx/
├── profile.json                          ← NEW: manifest of profiled packages
├── env                                   ← UPDATED: calls `ocx shell profile load`
├── env.ps1                               ← UPDATED: calls `ocx shell profile load`
├── objects/...
├── index/...
└── installs/...
```

The env file becomes static — it always calls `ocx --offline shell profile load` which dynamically reads `profile.json` and outputs shell exports at shell startup time (~20ms overhead, offline).

### Key Decisions

| Decision | Rationale |
|----------|-----------|
| `profile.json` manifest (not env file comments) | Clean separation of data and presentation. Enables future multiple-profile support. Schema-validatable. |
| Dynamic eval via `ocx shell profile load` | Always reads current metadata — no regeneration needed on `select` or version changes. ~20ms cost is acceptable. |
| Under `shell` subcommand group | Natural fit alongside `shell env` and `shell completion`. |
| No `--profile` flag on `install`/`uninstall` | Profiling implies `--select` which is redundant info. Profile management is a separate concern. |
| Supports `--candidate` and `--current` modes | Users choose resolution mode per package. Default `--current` (floating). `--candidate` pins to a specific tag. |
| Version field in manifest | Forward compatibility. Currently `1`. Schema generated by `ocx_schema`. |

## CLI Surface

```sh
ocx shell profile add cmake                    # default: --current
ocx shell profile add --current cmake           # explicit: floating pointer
ocx shell profile add --candidate cmake:3.28    # pinned to specific tag
ocx shell profile remove cmake                  # remove from profile
ocx shell profile list                          # show profiled packages with status
ocx shell profile load                          # output shell exports (called from env file)
```

## Profile Manifest (`$OCX_HOME/profile.json`)

```json
{
  "version": 1,
  "packages": [
    { "identifier": "ocx.sh/cmake", "mode": "current" },
    { "identifier": "ocx.sh/node:18", "mode": "candidate" }
  ]
}
```

The `version` field is a constant integer (currently `1`) for forward compatibility. The schema is generated by the `ocx_schema` crate alongside the metadata schema.

## Env File Templates

### POSIX (`$OCX_HOME/env`)

```sh
#!/bin/sh
# OCX shell environment — generated by ocx
_ocx_home="${OCX_HOME:-$HOME/.ocx}"
export PATH="${_ocx_home}/installs/ocx.sh/ocx/current/bin:$PATH"
_ocx_bin="${_ocx_home}/installs/ocx.sh/ocx/current/bin/ocx"
if [ -x "$_ocx_bin" ]; then
  eval "$("$_ocx_bin" --offline shell profile load 2>/dev/null)" 2>/dev/null || true
  eval "$("$_ocx_bin" --offline shell completion 2>/dev/null)" 2>/dev/null || true
fi
unset _ocx_home _ocx_bin
```

### Fish (`~/.config/fish/conf.d/ocx.fish`)

```fish
# OCX shell environment — generated by ocx
set -l _ocx_home (set -q OCX_HOME; and echo $OCX_HOME; or echo $HOME/.ocx)
fish_add_path --path "$_ocx_home/installs/ocx.sh/ocx/current/bin"
set -l _ocx_bin "$_ocx_home/installs/ocx.sh/ocx/current/bin/ocx"
if test -x "$_ocx_bin"
  "$_ocx_bin" --offline shell profile load --shell fish 2>/dev/null | source
  "$_ocx_bin" --offline shell completion --shell fish 2>/dev/null | source
end
```

### PowerShell (`$OCX_HOME/env.ps1`)

```powershell
# OCX shell environment — generated by ocx
$_ocx_home = if ($env:OCX_HOME) { $env:OCX_HOME } else { Join-Path $HOME ".ocx" }
if ($env:PATH -notlike "*$_ocx_home*") {
  $env:PATH = "$_ocx_home\installs\ocx.sh\ocx\current\bin;$env:PATH"
}
$_ocx_bin = Join-Path $_ocx_home "installs\ocx.sh\ocx\current\bin\ocx.exe"
if (Test-Path $_ocx_bin) {
  Invoke-Expression (& $_ocx_bin --offline shell profile load --shell powershell 2>$null)
  Invoke-Expression (& $_ocx_bin --offline shell completion --shell powershell 2>$null)
}
```

The env file is static — no regeneration needed on profile changes. Only `profile.json` changes.

All types derive `Serialize`/`Deserialize` and `JsonSchema` behind the `jsonschema` feature flag. Methods: `load()`, `save()` (atomic write), `add()`, `remove()`, `contains()`.

- [ ] **Step 1.3:** Register profile module

### Phase 2: CLI Subcommands

- [ ] **Step 2.1:** Create shell profile subcommand group
  - Files: `crates/ocx_cli/src/command/shell_profile.rs` (new)
  - Details: `enum ShellProfile { Add, Remove, List, Load }` following the `shell.rs` pattern.

- [ ] **Step 2.2:** Implement `shell profile add`
  - Files: `crates/ocx_cli/src/command/shell_profile_add.rs` (new)
  - Details: Accepts `ContentPath` (flatten, `--candidate`/`--current`), defaults to `--current`. Loads manifest, adds entries, saves.
  - **Validation**:
    - For `--candidate` mode: identifier **must** include a tag (since `candidates/{tag}` requires one). Error if tag is missing.
    - For `--current` mode: identifier needs registry + repo only (tag-agnostic).
    - Verify the corresponding symlink exists before adding. If `current` symlink is missing, error message should suggest `ocx install <pkg> --select` first. If `candidate` symlink is missing, suggest `ocx install <pkg>:<tag>` first.

- [ ] **Step 2.3:** Implement `shell profile remove`
  - Files: `crates/ocx_cli/src/command/shell_profile_remove.rs` (new)
  - Details: Accepts package identifiers (registry/repo). Loads manifest, removes matching entries, saves.

- [ ] **Step 2.4:** Implement `shell profile list`
  - Files: `crates/ocx_cli/src/command/shell_profile_list.rs` (new)
  - Details: Loads manifest, checks symlink status per entry. Reports via `api::data::profile::ProfileList`.

- [ ] **Step 2.5:** Implement `shell profile load`
  - Files: `crates/ocx_cli/src/command/shell_profile_load.rs` (new)
  - Details: Accepts `--shell` flag (or auto-detect). Loads manifest. For each entry: resolve via appropriate symlink mode, read metadata, use `ProfileBuilder` to generate shell exports. Print to stdout. Silently skip broken entries.

- [ ] **Step 2.6:** Register in shell subcommand group

## Risks

| Risk | Mitigation |
|------|------------|
| Shell startup overhead (~20ms per session) | Measured acceptable. `--offline` flag ensures no network. Failure-tolerant (stderr suppressed). |
| Broken profile entries after uninstall | `load` silently skips broken entries. `list` shows `Broken` status. |
| Profile manifest corruption | Atomic writes (temp file + rename). Version field for forward compat. |
| Multiple OCX instances modifying profile simultaneously | Not addressed in v1. File-level atomicity via rename is sufficient for typical usage. |

## Notes

- The env file becomes static after this change — it always delegates to `ocx shell profile load`. No regeneration needed on profile changes.
- `ocx shell profile load` reuses the same `ProfileBuilder` and metadata resolution as `ocx shell env`, making it functionally equivalent to a batched `shell env` driven by the manifest. All packages are resolved in-process (file reads + ProfileBuilder) — no additional subprocess spawns per package.

## Future Work (Out of Scope)

These are noted for future consideration but explicitly out of scope for this implementation:

- **Combined `ocx shell init` command**: The env file currently spawns two `ocx` subprocesses (one for `profile load`, one for `completion`). A future `ocx shell init` could combine both into a single subprocess invocation, halving the binary startup cost. Not needed until startup cost becomes a concern.
- **`shell profile load --json`**: Machine-readable output for tooling integration. Low priority.
- **Cache invalidation**: If `profile.json` hasn't changed since last load (mtime check), skip metadata reads and output cached result. Optimization for later.
- **Multiple named profiles**: `$OCX_HOME/profiles/dev.json`, `$OCX_HOME/profiles/ci.json` with `ocx shell profile switch dev`. The data model (version field, file-based manifest) supports this — just change the file path.
- **Lightweight `Context` for `load`**: `shell profile load` doesn't need `PackageManager`, `Index`, or OCI client — only `FileStructure` + file reads. If `Context::try_init` grows heavier over time, consider a lightweight init path. Currently `Context` with `--offline` is cheap enough.
