# Research: Shell Activation + Integration Patterns

**Date:** 2026-04-19  
**Purpose:** Deep analysis of shell activation options for project-level toolchain managers

---

## Recommendation

**Support two activation models. Do not add shims.**

1. **Primary: `ocx shell init <shell>` with prompt hook** (mise-style)
2. **Secondary: direnv `.envrc` generation** (`ocx generate direnv`)

---

## Industry Context

- **Trending**: Prompt-hook activation (mise, proto) + direnv integration as complementary approaches
- **Established**: `eval "$(tool env)"` for explicit activation; direnv for automatic cd-triggered activation
- **Emerging**: Nushell vendor autoload directory (landed 0.96.0, July 2024) — no eval needed
- **Declining**: Shell-forking (`poetry shell`) — Poetry removed it in 2.0, replaced with `eval "$(poetry env activate)"`

---

## Deep Analysis of Each Activation Option

### Option 1: `eval "$(ocx shell env)"` Pattern

**Who uses it**: Poetry 2.0, `nix print-dev-env`, cargo's env

**What it does**: User runs one-liner in terminal or sources in script; shell evaluates export statements

**OCX status**: Already works — `ocx shell env` exists, outputs shell-specific exports

**Limitation**: Entirely manual — developer must remember to run on every project entry, re-run after `ocx.toml` changes

**Verdict**: Good primitive, insufficient as user-facing interactive story. Keep as a building block.

### Option 2: `ocx shell init <shell>` with Prompt Hook (RECOMMENDED PRIMARY)

**Who uses it**: mise, proto, starship, atuin, zoxide — virtually every modern CLI tool

**Mechanism**: User adds `eval "$(ocx shell init bash)"` to `~/.bashrc`. OCX generates a shell-specific script that installs a hook. On every directory change, hook calls `ocx hook-env` which reads nearest `ocx.toml` and emits diff of PATH changes.

**Hook mechanisms by shell**:
| Shell | Mechanism | Init Line |
|-------|-----------|-----------|
| Bash | `PROMPT_COMMAND` tracking `$PWD` vs `$OLDPWD` | `eval "$(ocx shell init bash)"` |
| Zsh | `add-zsh-hook chpwd` + `add-zsh-hook precmd` | `eval "$(ocx shell init zsh)"` |
| Fish | `function __ocx_hook --on-variable PWD` | `ocx shell init fish \| source` |
| Nushell | Write to `$env.NU_VENDOR_AUTOLOAD_DIR` | `ocx shell init nushell` (writes file, no eval) |

**Performance**: Mise reports 2-5ms for hook-env evaluation in Rust. With mtime-based early-exit (hash the resolved `ocx.toml` path + mtime, skip if unchanged), shell prompt latency is negligible.

**PATH deduplication requirement**: Hook must maintain before/after diff. When user `cd`s out, hook emits unset + path-removal statements. Track injected entries in `$__OCX_ACTIVE_PATHS` shell variable.

**Security boundary**: Hook MUST NOT auto-install. Only exposes already-installed tools. Missing tools → silent skip or warning.

**OCX advantage**: `Shell::export_path()` and `Shell::export_constant()` in `shell.rs` already support all major shells. The output layer is done.

### Option 3: Direnv `.envrc` Generation (RECOMMENDED SECONDARY)

**What it is**: `ocx generate direnv` writes a `.envrc` file. User runs `direnv allow`. From then on, every `cd` auto-activates.

**Generated `.envrc`**:
```sh
# Generated by ocx — run 'direnv allow' to activate
watch_file ocx.lock
eval "$(ocx --offline shell-hook)"
```

**Security model**: `direnv allow` required per directory, per `.envrc` content change. Prevents malicious repos from auto-executing on `cd`.

**`--offline` is mandatory**: A `.envrc` that triggers network calls blocks shell startup. Project tools must already be installed.

**Limitation**: Cannot export shell aliases or functions — only env vars and PATH. OCX's use case (PATH entries + env vars from package metadata) fits perfectly.

**Integration**: nix-direnv, devenv, pixi all use this pattern. Established in enterprise and NixOS environments.

### Option 4: Shell Hook (cd hook) — Same as Option 2 Implementation

The hook mechanism IS what Option 2 generates. Not a separate user-facing option.

### Option 5: `ocx activate` Explicit Source

Similar to `source venv/bin/activate`. Lowest implementation cost, worst ergonomics. Requires remembering to run, no auto-deactivate.

**Verdict**: Implement as a thin wrapper (`ocx activate` → `eval "$(ocx --offline shell-hook)"`) for environments that cannot inject shell hooks (some CI runners). Not a primary path.

### Option 6: Shims — DO NOT IMPLEMENT AS PRIMARY

**Who uses it**: asdf, rbenv, pyenv

**Cost**: ~120ms per invocation (asdf), ~50ms (rbenv). For cmake/ninja/clang called thousands of times per build = catastrophic.

**`which` problem**: `which cmake` returns shim path, not real binary — breaks many tools.

**Legitimate use case**: Non-interactive contexts where prompt never fires (IDE language servers, CI scripts that don't source rc). Defer to after IDE integration is a stated requirement.

## Shell-Specific Implementation Notes

| Shell | PATH Deduplication | Notes |
|-------|-------------------|-------|
| Bash | Manual: `[[ ":$PATH:" != *":$dir:"* ]]` | No native chpwd; must diff `$PWD` vs `$OLDPWD` in PROMPT_COMMAND |
| Zsh | `$path` array with `(u)` flag | precmd catches non-cd changes to `$PWD` |
| Fish | `fish_add_path` is idempotent by design | Cannot use `eval`; must use `source` or pipe |
| Nushell | `if $dir not-in $env.PATH` | Autoload directory (0.96.0+); no eval needed |

---

## Hermeticity vs Ergonomics Tradeoff

| Activation Mode | Hermeticity | Ergonomics | Use For |
|----------------|-------------|------------|---------|
| `ocx exec --clean` | Maximum | Explicit per-invocation | CI, Bazel, GitHub Actions |
| `ocx exec` (no `--clean`) | Medium | Explicit per-invocation | Scripts needing host env |
| `ocx shell-hook` + direnv | Medium | Automatic on `cd` | Interactive dev sessions |
| `ocx shell profile` | Low (permanent) | Zero friction | Personal tool installs |

**The right framing**: not "how do we make activation hermetic" but "how do we make leakage predictable and reversible." The hook model gives both: activation scoped to directory tree, deactivation automatic on `cd` out, PATH changes trackable.

---

## Implementation Priority

1. `ocx shell init bash|zsh|fish` — covers ~95% of developer workstations
2. `ocx generate direnv` — `.envrc` generator (low effort, high security value)
3. `ocx shell init nushell` — autoload directory writer (Nushell growing fast)
4. Shims — defer until IDE integration is a stated requirement
