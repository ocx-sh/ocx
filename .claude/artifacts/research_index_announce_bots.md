# Research: Package-Index Announce Bots + Auto-Merge CI Patterns

**Date:** 2026-07-12
**Scope:** Validates and extends `adr_public_index_registry_indirection.md` (D4 "announce = doorbell", D5 bot merge policy, transport ladder v0→v2). This artifact is the formal external-precedent research the ADR flagged as missing ("informal discussion-level survey; no separate research artifact").

## Direct Answer

Every mature package index that regenerates entries from an authoritative source of truth converges on the **same two-tier split** the ADR already encodes in D5: *establishing trust for a new namespace* is always human-gated (PR + reviewer), while *routine refresh of an already-trusted entry* is bot-driven and auto-merges once CI/schema validation is green. The only real design freedom is **how the announce signal reaches the bot** and **how the bot authenticates the announcer** — and the industry is actively moving off long-lived static tokens (PATs) toward short-lived OIDC-derived credentials ("Trusted Publishing") for exactly that leg.

## Industry Context & Trends

- **Trending**: OIDC "Trusted Publishing" (OpenSSF Trusted Publishers standard) — npm GA July 2025, crates.io shipped 2025 (RFC 3691), PyPI/RubyGems already live. Replaces static PATs with short-lived, claims-scoped tokens minted per CI run. [npm changelog](https://github.blog/changelog/2025-07-31-npm-trusted-publishing-with-oidc-is-generally-available/), [crates.io RFC 3691](https://rust-lang.github.io/rfcs/3691-trusted-publishing-cratesio.html)
- **Established**: PR-based bot pipelines with label-gated auto-merge for routine changes, human review for anything that changes trust/ownership — Bazel Central Registry, Homebrew, winget all independently arrived here.
- **Emerging**: rate-limited public "refresh" endpoints decoupled from the PR-based registration flow (OpenTofu Registry RFC) — the announce endpoint is a doorbell, the PR flow is only for identity/trust establishment.
- **Declining**: fully-manual PR review for every version bump (doesn't scale — Homebrew/BCR/winget all automating it away); git-clone-the-whole-index protocols (crates.io's original git index, replaced by sparse HTTP).

## Key Findings

### 1. Bazel Central Registry (BCR)

Two cooperating mechanisms: `publish-to-bcr` (a reusable GitHub Actions workflow ruleset owners run in *their own* repo) opens the PR against the central registry, authenticated with a classic PAT (`BCR_PUBLISH_TOKEN`, recommended to belong to a dedicated machine user, not a human). **No automatic merging happens at that layer — human review is always required** ([publish-to-bcr README](https://github.com/bazel-contrib/publish-to-bcr)). The registry side then runs `bcr_presubmit.py`/`bcr_validation.py` CI; **manual review is mandatory when the module is new, when `presubmit.yml` changed materially, or for first-time contributors (CLA/workflow checks)** — otherwise a maintainer/BCR-maintainer approval + green presubmit is what merges it ([bazel-central-registry docs](https://github.com/bazelbuild/bazel-central-registry/blob/main/docs/README.md)). Net: **new module = PR + mandatory human approval; version bump from a listed maintainer = PR + approval-that-can-come-from-the-module-owner + green CI**, never fully bot-merged without any human in the loop.

### 2. winget-pkgs

Five-stage Azure DevOps validation pipeline (file validation → manifest content scan → catalog verification → installer security scan → post-validation labeling), orchestrated by `wingetbot`/`WinGetSvcWrapper` ([validation-pipeline.yaml](https://github.com/microsoft/winget-pkgs/blob/master/DevOpsPipelineDefinitions/validation-pipeline.yaml)). Critically: **`Validation-Completed` does not auto-merge by itself** — the bot posts a comment requiring a community moderator's approval *unless* the PR already carries `Moderator-Approved`, `Validation-Metadata`, or `Publisher-Verified` ([DeepWiki PR lifecycle](https://deepwiki.com/microsoft/winget-pkgs/5.1-auto-merge-and-validation-completion)). So the real auto-merge gate is a **pre-established trust label** (verified publisher), not "is this just a version bump." Dynamic-scan-only completions never auto-merge — a Microsoft employee merges manually regardless.

### 3. Homebrew autobump / BrewTestBot

BrewTestBot scans the autobump list every 3 hours and opens PRs for new upstream versions automatically ([Autobump docs](https://docs.brew.sh/Autobump)). It **auto-merges** once (a) a maintainer with write access approves, and (b) CI is green — no further human action needed after approval ([BrewTestBot-For-Maintainers](https://docs.brew.sh/BrewTestBot-For-Maintainers)). **New-formula PRs are explicitly excluded from this fast path** (along with `automerge-skip`/`autosquash`-labeled PRs), forcing manual `brew pr-publish`/`brew pr-pull`. Packages opt out of autobump entirely via `no_autobump!` (with a mandatory reason) — the closest prior art to OCX's per-package `status`/exclusion concept.

### 4. Go module proxy / checksum database (sumdb)

No PR model at all. `index.golang.org` is an append-only transparency log; entries are created **on first request**, not announced by publishers — "announce = doorbell, and the doorbell can't lie" is Go's actual design philosophy, already the direct inspiration named in the ADR's D4 row ([Go sumdb proposal](https://go.googlesource.com/proposal/+/master/design/25530-sumdb.md)). Trust model is **detection over prevention**: clients cryptographically verify log consistency; a compromised entry is provably detectable after the fact rather than blocked before merge. This is viable specifically because Go never trusts the *announcer's* claims — it always re-derives content from the module's own VCS host, exactly OCX's stated principle for OCI truth. crates.io independently converged on the same shape at scale: it dropped its git-PR index entirely in favor of a server that writes the sparse HTTP index directly on `cargo publish` — **zero PRs, zero bots, one authoritative writer** ([Cargo Book — registry index](https://doc.rust-lang.org/cargo/reference/registry-index.html), [sparse registry stabilization](https://github.com/rust-lang/cargo/pull/11224)).

### 5. OpenTofu Registry — closest structural analog to ocx-rs/index

Registration of a *new* provider/module/signing-key is PR-based and always human-reviewed (explicit callout: "you have to wait for the PR with your malicious key to be approved and merged" — the review *is* the security control). But **routine version discovery needs no PR at all**: the registry polls GitHub releases-RSS/`git ls-remote` for already-registered repos, and separately offers an **optional rate-limited public "refresh" endpoint** publishers can hit to force an immediate re-scan ([RFC #724](https://github.com/opentofu/opentofu/issues/724), [Building the OpenTofu Registry](https://opentofu.org/blog/building-the-opentofu-registry/)). This is the clearest existing precedent for exactly the split ocx-rs/index wants: **PR = trust establishment (rare, human), refresh = doorbell (frequent, rate-limited, no PR)**.

### 6. Announce-endpoint auth mechanics (GitHub-specific)

`repository_dispatch`/`workflow_dispatch` are **never anonymous** — GitHub requires a token (repo-scoped PAT, GitHub App installation token, or `GITHUB_TOKEN` from within the same repo) to call the dispatch API; there is no such thing as a truly public unauthenticated dispatch endpoint on GitHub ([Triggering a workflow](https://docs.github.com/actions/using-workflows/triggering-a-workflow)). GitHub App tokens are the recommended cross-repo mechanism because they're ephemeral (1h) and scoped to specific repos/permissions rather than bound to a human identity ([GitHub Actions token guidance](https://github.blog/changelog/2022-09-08-github-actions-use-github_token-with-workflow_dispatch-and-repository_dispatch/)). Rate limiting on abuse: GitHub's secondary rate limits (900 points/min, no >500 content-generating requests/hr) already throttle runaway dispatch storms; recommended mitigation for bulk triggers is exponential backoff + the `Retry-After` header ([REST API rate limits](https://docs.github.com/en/rest/using-the-rest-api/rate-limits-for-the-rest-api)). For **untrusted public input specifically** (issue-based triggers), GitHub's own mitigation is workflow-run approval gating for first-time/external contributors ([Approving workflow runs from forks](https://docs.github.com/en/actions/how-tos/manage-workflow-runs/approve-runs-from-forks)) plus the IssueOps pattern of parsing structured issue-form bodies rather than free text, always being careful that `${{ }}` interpolation of attacker-controlled fields isn't shell-injected ([GitHub Security Lab — untrusted input](https://securitylab.github.com/resources/github-actions-untrusted-input/)).

## Design Patterns Worth Considering

- **Trust-once-then-doorbell** (OpenTofu, Go, crates.io-after-migration): identity/ownership is proven exactly once (PR review or publish-time auth), every subsequent update re-derives content from the source of truth and skips human review entirely. This *is* the ADR's D4/D5 already — the research here is confirmation, not correction.
- **Label-as-trust-cache** (winget's `Publisher-Verified`/`Moderator-Approved`, Homebrew's absence of `new-formula`): don't re-derive "is this publisher trusted" on every PR — cache the trust decision as a durable label/field (OCX's `owners` field in D6 already plays this role) so the bot's auto-merge check is a cheap lookup, not a re-review.
- **OIDC Trusted Publishing over static PATs**: the industry-standard replacement for the `BCR_PUBLISH_TOKEN`-style secret every announcer currently has to hold. Directly relevant to the ADR's Ground Setup TODO ("Announce doorbell credential: fine-grained PAT... or a small GitHub App later") — see Recommendation below.
- **Re-derive, never trust payload content** (Go sumdb, D3/D4's pointer-only entries): the announce payload should carry nothing but a package/version identifier; every field written to the index file must come from a live OCI registry read in the bot job, never from the dispatch body. OCX already designed this in; it's the single most important anti-tamper property across every precedent studied.
- **Direct-write over PR-for-everything at scale** (crates.io's abandonment of its git-PR index): once refresh volume is high, opening-then-auto-merging a PR per version bump is wasted motion and risks GitHub's abuse-detection on repos generating "hundreds of PRs... practically at the same moment." A bot-authored direct commit to `main` (protected branch, one CI identity exempted) is the natural next step once volume justifies it — not needed at 42–400 packages, but worth flagging now rather than rediscovering later.

## Pitfalls Confirmed Across Precedents

- **Announce/registry propagation race**: OCI registry writes are strongly consistent per-repo (manifest PUT only returns after commit), but the *bot's own read* immediately after a dispatch can still race a not-yet-visible cache layer on some registries/mirrors. Every reconciliation-based system studied (Go, OpenTofu, Homebrew) treats the periodic full rescan as the correctness backstop, not the happy path fetch. Recommend the bot job retry-with-backoff on manifest fetch (a handful of attempts over ~1 minute) before giving up.
- **Stale-entry reconciliation**: never hand-patch an index file; always regenerate the whole per-package entry from OCI truth and diff (already D4's "nightly reconcile cron"). If a nightly reconcile detects that an *already-published* version's digest changed, that's an integrity signal (potential tamper/registry compromise), not a routine update — every precedent that discussed this (Go's transparency log, BCR's `unstable_url` skip-check) treats immutable-version mutation as an anomaly requiring a human flag, not a silent auto-fix.
- **PR queue flooding**: every registry that scaled past hobby-size (crates.io, winget, BCR) eventually pushed non-trust-affecting updates *off* the PR path entirely, because a PR-per-version-bump model degrades (abuse-detection triggers, review queue backlog) well before it hits tens of thousands of packages. OCX's 42→hundreds trajectory doesn't need this yet, but the "auto-merge" leg of D5 should be scoped from day one so it can become a direct-commit bot without a redesign.

## Recommendation

**The ADR's D4/D5 design is already correct and matches every precedent studied — do not redesign it.** This research's value is in refining the one part the ADR left open: the **announce-credential mechanism** (Ground Setup currently plans "fine-grained PAT... or a small GitHub App later").

Concretely, for `ocx-rs/index`:

1. **Keep the PR-for-new-package / bot-direct-refresh split exactly as D5 states.** Every precedent (BCR, winget, Homebrew, OpenTofu) independently re-derives this same boundary; it is not a place to innovate.
2. **v1 announce transport: skip the GitHub App, use plain `repository_dispatch` + a fine-grained PAT per registered namespace** (issued during the namespace's human-reviewed registration PR, stored as a secret in the *publisher's* repo — same shape as BCR's `BCR_PUBLISH_TOKEN`). A GitHub App adds hosting/webhook infrastructure OCX doesn't need at 42–400 packages and conflicts with the "zero production services owned by us" driver already in the ADR — boring tech wins here.
3. **Plan the v2 upgrade path to OIDC Trusted Publishing now, in the Deferred section, not as an afterthought.** Instead of every publisher holding a static PAT forever, `ocx-rs/index` should eventually verify a GitHub Actions OIDC ID token (JWT from `token.actions.githubusercontent.com`) asserting the announcer's own repo/workflow identity against the `owners` field already stored in the entry schema, then dispatch internally with its own short-lived credential — mirroring exactly what crates.io/npm now do for `cargo publish`/`npm publish`. This removes the last long-lived secret from the announce path and costs nothing until namespace count justifies building the tiny verifier (a Worker or a reusable Action), so it belongs in Deferred/Out of Scope alongside signing/provenance, not in v1.
4. **Nightly reconcile must treat digest-mutation on an already-published version as a hard-stop anomaly, not a silent overwrite** — add this explicitly to D4's reconcile-cron description; none of the systems studied auto-heal an integrity violation, they all flag it for a human.
5. **When refresh volume eventually makes PR-then-auto-merge itself the bottleneck** (crates.io's own lesson), the upgrade is a bot-authored direct commit to `main` for the refresh path only — keep this noted as the trigger condition in the Deferred section rather than building it now (YAGNI at current scale).
