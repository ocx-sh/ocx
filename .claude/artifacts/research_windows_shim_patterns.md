# Research: Embed + Rollout Patterns for Windows Shim (issue #66)

**Date:** 2026-05-18 | **Axis:** Design patterns | **Expires:** 2027-05-18

## Direct Answer

Commit prebuilt signed shim blobs to git (uv/pixi model), embed via arch-gated
`include_bytes!` in a new `crates/ocx_lib/src/shim.rs` (no `build.rs`). At
install time copy bytes verbatim → `<name>.exe`, write one-line `<name>.shim`
sidecar (pkg_root). Authenticode survives verbatim copy. Rollout = additive:
write `.exe`+`.shim`+`.cmd` together; `.exe` wins PATHEXT.

## Findings

1. **Chicken-egg (sign in CI vs `include_bytes!` at compile):** industry solves
   by committing prebuilt signed blobs (uv `uv-trampoline/trampolines/*.exe`,
   pixi `build-trampoline` script). Main build embeds committed blob; blob
   refreshed in dedicated PR when shim source changes. Avoids `build.rs`
   orchestration and CI ordering complexity.
2. **Authenticode survives copy/extract unconditionally.** Hash excludes
   CheckSum, Cert-Table Data-Directory entry, Cert Table itself. `include_bytes!`
   + verbatim `fs::write` → `signtool verify /pa` still passes. Pitfall: NEVER
   `strip` after signing (modifies PE sections → invalidates). Strip before sign.
3. **Sidecar contract:** one line = absolute pkg_root. NOT target path (would
   lose `ocx launcher exec` resolution indirection), NOT digest, NOT config
   flags (env-var territory). Plaintext, UTF-8 no BOM. Tamper boundary unchanged:
   `validate_package_root` in `launcher exec` already validates pkg_root inside
   `$OCX_HOME/packages/` — same as today's `.cmd`.
4. **Rollout — Option A (write all three, recommended):** launchers are
   install-local, regenerated by `ocx install`/`select`; OCX's compat guarantee
   covers OCI manifests/metadata only, NOT launchers (confirmed). Default PATHEXT
   makes `.exe` win over `.cmd`. Zero migration, graceful fallback. Reject
   one-shot-rebuild (orphaned `.cmd` + PATHEXT-order confusion) and flag-gating
   (needless complexity).
5. **CI sequencing:** Phase1 build shim per arch → Phase2 sign (serial) →
   Phase3 parent build embeds signed blob + signs parent. cargo-dist 0.31 has no
   cross-compile; use native runners or commit-blob model. cargo-dist regen
   (`release.yml`) leaves hand-authored workflows alone; wire shim build via
   `plan-jobs` in `dist-workspace.toml`.
6. **Dev builds:** committed signed release blobs double as dev stubs; non-Windows
   builds use `#[cfg(not(windows))] SHIM_BYTES: &[u8] = &[]` and skip emission.

## Pitfalls

Strip-after-sign invalidates; `.exe` path relative to source file →
`crates/ocx_lib/src/shims/`; sidecar is not a security boundary;
SmartScreen reputation lags even when signed; cargo-dist regen overwrites
`release.yml` only.

## Recommendation

Pattern A committed-blob + arch-gated `include_bytes!`, no `build.rs`. Add 4th
file in `launcher::generate()` on Windows: `<name>.exe` (SHIM_BYTES) +
`<name>.shim` (pkg_root sidecar), keep `<name>.cmd`. Atomic write-then-rename;
sidecar written after `.exe`. x86_64 v1, aarch64 follow-on.
