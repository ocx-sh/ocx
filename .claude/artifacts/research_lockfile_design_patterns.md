# Research: Group/Environment Composition Models and Activation Patterns

**Date:** 2026-04-19  
**Purpose:** Inform `ocx.toml` group model and `ocx exec` + shell activation design

---

## Group Model Recommendation

**Adopt a two-level additive model (simplified pixi-inspired):**

```toml
[tools]
cmake = "3.28"       # implicit "default" group — always included
ninja = "1.11"

[group.ci]
actionlint = "1.7"
hadolint = "2.12"

[group.java]
gradle = "8.5"
jdk = "21"
```

**Rules:**
1. `[tools]` is the implicit base group — always included in any activation
2. Named groups in `[group.<name>]` are **additive** on top of `[tools]`
3. Groups can include other groups: `inherits = ["ci"]` for nested composition (v2 feature)
4. **Conflict resolution**: fail-fast — same tool at different versions across groups = config error
5. Lock file pins digests for ALL tools across ALL groups in a single flat table; a group is just a named filter

## Group Model Comparison

| Tool | Group Syntax | Additive? | Partial Activation | Conflict Resolution |
|------|-------------|-----------|-------------------|---------------------|
| **pixi** | `[feature.<name>]` + `[environments]` | Yes — features union | Yes — `pixi run -e <env>` | None — solver fails |
| **mise** | `mise.<env>.toml` overlay files | Override (not union) | No — all tools in merged config active | Last profile wins |
| **Poetry** | `[tool.poetry.group.<name>]` | Yes — all groups in single solve | Yes — `--with`/`--without`/`--only` | Single global solve, fails on incompatibility |
| **devcontainer** | `"features": { ... }` in JSON | Yes — all features compose | No — all active | Last-write-wins |
| **nix flakes** | `devShells.<name>` in `flake.nix` | N/A — each shell independent | Yes — `nix develop .#<name>` | Build-time isolation |

**Why not mise's overlay:** File-level overlays are "dev vs prod" environment switching, not composable subsets within one file.

**Why not Poetry's model:** Requires a global solver; OCX uses digest pinning, so "all groups must be compatible" is trivially satisfied.

## Activation Model Recommendation

**Three tiers, prioritized:**

### Tier 1 — Hermetic exec (primary, already implemented)
```bash
ocx exec [--group ci] cmake -- cmake -B build
```
- `--group` flag selects which groups supply tools
- Lock resolves digests; `find_or_install_all` pulls missing tools
- `--clean` strips ambient PATH for full hermeticity
- **Use case**: CI, Bazel, GitHub Actions — the OCX product center of gravity

### Tier 2 — direnv integration (recommended for interactive use)
```bash
# .envrc — generated by ocx generate direnv
watch_file ocx.lock
eval "$(ocx --offline shell-hook)"
```
- `ocx shell-hook` outputs env var exports for the current project's `ocx.toml` (analogous to `pixi shell-hook`)
- direnv handles the `cd` hook, directory exit cleanup, and security model (`direnv allow`)
- `--offline` flag is mandatory — interactive activation must never trigger network I/O
- **No custom shell hook needed** — delegate to direnv entirely

### Tier 3 — Shell profile (already implemented, for persistent tools)
```bash
ocx shell profile add cmake:3.28
```
- For tools the user wants globally available on their dev machine
- Unchanged from current behavior

## Activation Model Comparison

| Mechanism | Hermeticity | Ergonomics | Security | Shell Compat | Impl Cost |
|-----------|-------------|------------|----------|--------------|-----------|
| `eval "$(ocx shell env)"` | None (leaks) | Low (manual) | Neutral | Bash/Zsh easy; Fish different | Very Low (exists) |
| `ocx shell init <shell>` + prompt hook | None | High (automatic) | Medium (no per-project opt-in) | All shells, different mechanism | Medium |
| direnv `.envrc` generation | None | High (cd-aware, auto after `allow`) | High (`direnv allow` per project) | All via direnv | Low |
| `ocx activate` (explicit source) | None | Low (manual, no auto-deactivate) | Neutral | Same as eval | Very Low |
| Shims | High per-binary | Medium (PATH-based) | Low | All — just a binary | High |

## Shell Activation Deep Dive

### Why direnv is the right primary interactive model

1. **Security**: `direnv allow` per project prevents malicious repo auto-execution
2. **ergonomics**: automatic on `cd`, deactivates on `cd` out
3. **OCX already has the output layer**: `Shell::export_path()` and `Shell::export_constant()` in `shell.rs` already generate correct syntax for bash/zsh/fish/PowerShell/Elvish/Ksh/Ash/Dash/Batch
4. **`ocx shell-hook` is minimal work**: thin wrapper that reads `ocx.toml`, resolves groups, calls existing env generation

### Why NOT shims

- Asdf shims: ~120ms per invocation; mise PATH activation: ~5ms
- For build tools (cmake, ninja, clang), shims invoked thousands of times per build = catastrophic
- `which cmake` returns shim path, not real binary — breaks many tools

### Why NOT a custom `ocx activate` shell hook

- Implementing prompt hooks for bash/zsh/fish/nushell/PowerShell is months of work
- direnv already solved this — delegate instead

### Security boundary for ANY auto-activation

**Critical**: The activation hook MUST NOT auto-install. It reads `ocx.toml` and prepends PATH for already-installed packages only. If a package is not installed, the PATH entry is silently skipped (or warns). Auto-install on `cd` is a supply-chain vulnerability.

## Pull-on-Demand vs Pre-Install

| Tool | Install Timing | Missing Tool UX |
|------|---------------|-----------------|
| mise | Eager (`mise install`) OR lazy via `mise exec`/`mise run` auto-install | Shows "download... verify... installed" |
| pixi | Eager — `pixi install` before any activation | Progress output during install |
| asdf | Explicit only — no lazy install | Silent command-not-found |
| OCX current | Lazy via `find_or_install_all()` in `exec` | (existing behavior) |

**OCX should keep the lazy-via-exec model** — `find_or_install_all()` is already the right behavior.
